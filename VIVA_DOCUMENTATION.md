## 1. Project Overview & Architecture

### High-Level Architecture
SmartCampus follows a classic, decoupled **Client-Server Architecture** utilizing the **MVVM (Model-View-ViewModel)** design pattern on the frontend and an asynchronous, non-blocking **RESTful API** backend. 

- **Frontend (Client)**: Native Android Application built with Kotlin and XML.
- **Backend (Server)**: Ktor framework built on Kotlin, running on a Netty engine.
- **Database**: PostgreSQL (Relational Database Management System) interacted with via JetBrains Exposed ORM.

### Why this Architecture?
- **Decoupling**: Separating the frontend from the backend allows for independent scaling. If the mobile app goes viral, you can scale the Ktor backend horizontally without touching the mobile codebase.
- **Security**: The database is never exposed directly to the internet. The backend acts as a secure proxy, authenticating requests via JWT (JSON Web Tokens).

## 2. Technology Stack: Under the Hood

### 2.1 Kotlin (Frontend & Backend)
**What happens under the hood:** Kotlin is statically typed and compiles down to Java Bytecode (`.class` files), which is then executed by the JVM (or Dalvik/ART on Android).
**Why chosen:** 
- **Null Safety**: Eliminates the billion-dollar mistake (`NullPointerException`) at compile time.
- **Coroutines**: Instead of heavy OS-level threads, Kotlin uses Coroutines (lightweight threads). Under the hood, a coroutine is a state machine generated by the Kotlin compiler. It allows suspending functions without blocking the underlying thread, enabling massive concurrency (perfect for a Ktor backend handling thousands of students checking placements simultaneously).

### 2.2 XML (Frontend UI)
**What happens under the hood:** XML layout files are compiled into a binary format (binary XML) by the Android Asset Packaging Tool (AAPT2) during the build process. At runtime, the `LayoutInflater` uses an XML Pull Parser to parse this binary XML and instantiate the corresponding Java/Kotlin View objects in memory.
**Why chosen:** Separation of concerns. UI declaration is kept completely separate from business logic, making it easier to read, maintain, and support multiple screen sizes.

### 2.3 Ktor (Backend Framework)
**What happens under the hood:** Ktor does not use the traditional Java Servlet API (like Spring Boot). Instead, it runs on top of asynchronous engines like Netty. It heavily relies on Kotlin Coroutines to handle HTTP requests asynchronously.
**Why chosen:** Extremely lightweight, fast startup time, and idiomatically Kotlin. It doesn't rely on heavy reflection or annotations, meaning lower memory footprint and faster execution.

### 2.4 PostgreSQL & JetBrains Exposed ORM
**What happens under the hood:** PostgreSQL stores data in structured tables using B-Tree indexes for fast retrieval. JetBrains Exposed acts as an Object-Relational Mapper (ORM). Instead of writing raw SQL strings, Exposed uses Kotlin's type-safe DSL (Domain Specific Language) to generate optimized SQL queries at runtime.
**Why chosen:** PostgreSQL is ACID-compliant, ensuring data integrity for critical placement data. Exposed prevents SQL Injection attacks by safely parametrizing queries and provides compile-time safety (if a column name changes, the code won't compile, preventing runtime crashes).

## 3. Folder Structure & File Purpose

### Frontend (`SmartCampusApp/app/src/main/`)
- `java/com/smartcampus/app/api/`: 
  - `ApiClient.java`: Configures Retrofit and OkHttp. Purpose is to establish a singleton network client with the `BASE_URL`.
- `java/com/smartcampus/app/ui/`: Contains all Activities/Fragments (e.g., `LoginActivity`, `StudentDashboardActivity`). Handles UI lifecycle events (`onCreate`, `onResume`).
- `res/layout/`: Contains all XML files (`activity_login.xml`, `item_job.xml`).
- `build.gradle`: Manages dependencies (Retrofit, Room, Hilt) and build configurations.

### Backend (`SmartCampusBackend/src/main/kotlin/com/smartcampus/backend/`)
- `Application.kt`: The entry point. Initializes Ktor, Netty, Database connection, and JWT authentication plugins.
- `routes/`: Controllers. E.g., `JobRoutes.kt`, `SkillRoutes.kt`. They define API endpoints (`/jobs`, `/skills`), extract request parameters, and delegate to Services.
- `service/`: Business Logic Layer. E.g., `RecommendationService.kt`, `SkillService.kt`. This is where algorithms and database transactions live.
- `models/`: Database schema definitions.
  - `Tables.kt`: Defines Exposed `Table` objects (e.g., `object Users : Table("users")`). Maps Kotlin objects to PostgreSQL tables.
- `dto/`: Data Transfer Objects. Data classes specifically formatted for JSON serialization to send back to the frontend (hiding sensitive DB fields).

## 4. Database Design & Schema

The database is highly relational. **Primary Keys** (`id`) and **Foreign Keys** (`user_id`, `student_id`) enforce referential integrity.

### Core Tables (`Tables.kt`)
1. **Users**: The master authentication table. Columns: `id`, `name`, `email`, `password` (hashed), `role` (STUDENT, RECRUITER, ADMIN).
2. **StudentProfiles**: 1-to-1 relationship with `Users`. Stores `cgpa`, `semester`, `branch`, `placementStatus`.
3. **Skills & StudentSkills**: Many-to-Many resolution. `Skills` holds unique skills (e.g., "Python"). `StudentSkills` links a profile to a skill with a `proficiencyLevel`.
4. **Jobs & JobApplications**: `Jobs` holds company postings. `JobApplications` links `student_id` to `job_id` with a `status` (APPLIED, SHORTLISTED, SELECTED).
5. **TrendingSkills**: Stores analytical data scraped from APIs (demand_score, region, growth_rate).

### Why this design?
3NF (Third Normal Form) normalization. We separate `Users` (auth data) from `StudentProfiles` (metadata) to ensure that if a Recruiter logs in, they don't have empty student-specific columns (like CGPA) taking up space. It prevents data anomalies and redundancy.

## 6. Algorithms and Heuristics

### 6.1 Personalized Skill Recommendation Algorithm
**Location:** `RecommendationService.kt`
**Complexity:** O(N * S) where N is trending skills and S is student's known skills.
**How it works:** It uses a multi-factor Weighted Heuristic Scoring Model. It assigns a score to every trending skill based on the student's unique profile.

**The Formula:**
`Final Score = (2.0 * RDS) + (2.0 * SRS) + (1.5 * DMS) + (1.5 * SGS) + (1.0 * IGS)`

- **RDS (Region Demand Score)**: Is it demanded in their preferred city? (0 or 1)
- **SRS (Semester Relevance Score)**: Is the student in the right semester to learn this? (1.0 for perfect match, 0.8 for adjacent, 0.2 for off-target).
- **DMS (Domain Match Score)**: Does it match their interest (e.g., AI vs Web Dev)? (0 or 1).
- **SGS (Skill Gap Score)**: Do they already know it? (1.0 if unknown, 0.0 if advanced - we don't recommend what they already mastered).
- **IGS (Industry Growth Score)**: Macro-market growth rate. (>20% = 1.0, >10% = 0.7).

**Reasoning:** A simple database query isn't enough. A student in Semester 1 shouldn't be recommended "Advanced Kubernetes" even if it's trending. This algorithm contextually bridges the gap between global trends and individual student readiness.

### 6.2 Skill Gap Analysis Algorithm
**Location:** `SkillService.kt` -> `getSkillGapAnalysis()`
**Logic:** Set Theory. 
`MissingSkills = TopDemandedMarketSkills - StudentKnownSkills`
`Readiness Percentage = (Count(MatchedSkills) / Count(TopDemandedMarketSkills)) * 100`
It calculates the exact percentage of market readiness and returns the critical missing nodes.

## 7. End-to-End Workflow: Job Application

1. **User Action**: Student taps "Apply" on a job in `activity_job_detail.xml`.
2. **Frontend Layer**: `JobDetailActivity` grabs the `jobId` and fires a Retrofit POST request to `BASE_URL/jobs/{jobId}/apply`. It attaches the JWT token in the Authorization header.
3. **Routing Layer**: Ktor receives the request in `JobRoutes.kt`. The `authenticate("auth-jwt")` block intercepts it, verifies the token signature, and extracts the `userId`.
4. **Service Layer**: `JobService.applyForJob(userId, jobId)` is called.
5. **Database Layer**: JetBrains Exposed opens a `transaction {}`. It checks if the student already applied (preventing duplicates). It then `INSERT`s a new row into `JobApplications` with status `APPLIED`.
6. **Response**: The transaction commits. Ktor serializes a `MessageResponse("Applied successfully", true)` into JSON and sends it back with HTTP 200 OK.
7. **UI Update**: Retrofit receives the 200 OK. The Android app shows a Toast message "Successfully Applied!" and updates the button to "Applied".

## 8. Likely Viva Questions & Strong Defenses

### Q1: Why did you choose Kotlin/Ktor over Java/Spring Boot for the backend?
**Strong Answer:** "I chose Ktor because it is built natively on Kotlin Coroutines. Unlike Spring Boot's traditional Thread-per-request model, which consumes massive OS memory under heavy load, Ktor's asynchronous non-blocking I/O allows a single thread to handle thousands of connections. Since a campus placement app experiences massive traffic spikes (e.g., when a Google drive is announced), Ktor ensures the server won't crash from thread exhaustion."

### Q2: How did you secure the API? What happens if someone steals a token?
**Strong Answer:** "The API is secured using stateless JWT (JSON Web Tokens). The token contains the user's ID and Role, cryptographically signed using HMAC with a secret key (`jwt.secret`). If someone alters the token to elevate their privileges to 'ADMIN', the cryptographic signature will become invalid, and Ktor will instantly reject it with a 401 Unauthorized. To mitigate token theft, tokens have a strict expiration time (`expirationMs`), minimizing the window of vulnerability."

### Q3: Your Recommendation Algorithm uses O(N) complexity. Will this scale if you have a million skills?
**Strong Answer:** "While the theoretical complexity is O(N), the actual implementation groups and filters the data at the database level first. We limit the input to the Top trending skills, meaning N is a small constant (e.g., top 100 skills). Furthermore, since trending skills don't change every second, this data can be easily cached using Redis in the future to completely bypass the computation for subsequent requests, making it effectively O(1) for cached reads."

### Q4: Why use a Relational Database (PostgreSQL) instead of NoSQL (MongoDB)?
**Strong Answer:** "A placement system is fundamentally relational. A Student applies to a Job, posted by a Recruiter, who works for a Company. Using NoSQL would require massive data duplication or slow application-level joins. PostgreSQL enforces strict ACID compliance. During a placement drive, if a student applies, we must guarantee consistency—either the application is saved entirely, or it fails. We cannot risk eventual consistency where an application is 'lost' in the void."

### Q5: What is the purpose of the `Retrofit` library you used? Why not use standard `HttpURLConnection`?
**Strong Answer:** "Standard `HttpURLConnection` requires extensive boilerplate: managing background threads, manually reading input streams, and parsing raw strings into JSON. Retrofit abstracts all of this. It uses annotations to define routes, automatically handles background execution via OkHttp, and uses Gson to automatically serialize/deserialize JSON strings directly into Kotlin Data Classes. It drastically reduces bug surface area and improves code maintainability."
